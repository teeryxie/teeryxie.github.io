# 从实时对话到岗位 Agent：AI 面试官、服务工作与“替代人工”的证据

实时语音模型最初解决的是体验问题：让 AI 听得更快、说得更自然、允许用户打断。下一阶段的问题已经变得更具体：当模型能够持续对话、调用工具、维护流程状态并生成评价时，它是否可以接替原本由人工完成的互动工作？例如主持招聘面试、完成客服工单、采集软件需求、进行医疗问诊、做销售资格筛选或承担教育陪练。

答案不是简单的“可以”或“不可以”。目前确实已有生产系统每天自动执行上千场面试，也有模型在受控医疗对话实验中超过医生对照组；但同一批研究也表明，AI 面试官可能追问浅、漏掉大量隐含信息，自动 judge 与真人满意度几乎不相关，客服 agent 在重复执行同一任务时高度不稳定，语音系统被打断后甚至无法回到正确流程节点。

因此，这篇文章不讨论抽象的“AI 会不会取代人类”，而是提出一个更严格的问题：**实时交互 Agent 在什么任务、什么风险和什么证据条件下，已经能够主导流程；又在什么条件下，只适合做人工副驾驶？** 我的核心判断是：当前进展正在从“生成一段回复”走向“接管一个结构化 episode”，但距离“接管一个岗位”仍有明显差距。

![Automation ladder](/images/blog/work-agent-automation-ladder.svg "图 1：实时交互系统从提供建议、主导单次会话，逐渐走向结构化流程接管；岗位级替代还需要覆盖例外、责任与长期结果。")

## 1. “替代人工”至少有四个不同层级

大量讨论把自动生成问题、自动主持对话和自动做最终决策混成同一件事。为了避免夸大，我把人机互动工作的自动化分成四级。

- **L1 辅助人工**：AI 转录、总结、推荐下一问题、检索资料或给出评分草稿；人类仍决定问什么、说什么和采取什么行动。
- **L2 主导单次互动**：AI 独立主持一次面试、客服通话或问诊，能动态追问和生成记录；人类在会后审核结果。
- **L3 接管结构化流程**：AI 不只聊天，还读取数据库、更新工单、预约、发送材料、处理中断和例外，并在达到升级条件时转人工。
- **L4 岗位级代理**：AI 覆盖该岗位长期面对的任务分布、模糊目标、异常情况、跨系统协作、伦理责任和绩效结果，只在少数情形请求监督。

目前最成熟的是 L1，部分客服、招聘筛选和预约场景已经进入 L2-L3。公开研究尚未证明任何通用实时 Agent 稳定达到 L4。原因是岗位不是一组平均任务，而是包含长尾事件：身份争议、情绪冲突、规则例外、权限变更、欺诈、合规申诉，以及“用户说的”和“用户真正需要的”不一致。

这种分级也解释了为什么部署规模不是替代证据。一个系统每天主持 1500 场面试，能够证明吞吐、成本和基本流程已经工程化；它不能自动证明追问质量、公平性、评分效度和最终招聘结果优于人工。

## 2. 技术演进：为什么现在才开始接近岗位代理

传统语音客服和 IVR 很早就能根据按键或固定 intent 路由，但它们无法应对开放表达。2010 年代的 task-oriented dialogue system 引入 dialog state tracking、slot filling 和 API action，能够预订餐厅或查询订单，却依赖有限领域和人工编写 ontology。

大模型带来了三个变化。

第一，模型可以在没有固定话术树的情况下理解长表达、总结上下文并生成追问。第二，function calling 和 computer-use agent 让语言输出连接到 CRM、日历、邮件、浏览器和企业 API。第三，GPT-4o、Gemini Live、Nova Sonic、Qwen-Omni、Moshi 等实时语音或 omni 系统降低了交互延迟，使 Agent 可以像坐席一样边听边处理，而不是等完整录音结束后离线分析。

但真正推动“接替工作”的并不是语音自然度，而是评测对象发生了变化：从单轮答案正确率，转向数据库最终状态、流程完成率、多次运行一致性、中断恢复和真人结果。`tau-bench`、`tau2-bench`、CRMArena、OSWorld、ReqElicitGym、YIELD 与 IHBench 都是在测一个共同问题：模型能否在持续互动中把信息变成正确、合规、可重复的行动。

## 3. AI 面试官：最接近规模化部署，也最容易混淆目标

“面试”至少包括三类不同任务。

- **练习型面试**：帮助候选人准备，目标是覆盖问题、给出反馈和改善表达。
- **信息采集型面试**：招聘、需求分析、新闻调查或质性研究中，目标是通过追问获得完整、可靠的信息。
- **评估与决策型面试**：根据证据给候选人打分、排序、筛选甚至拒绝，直接影响机会分配。

前两类主要考验交互和信息获取；第三类额外涉及测量效度、公平、申诉和法律责任。一个适合模拟练习的数字人，并不因此具备替企业做录用决定的资格。

### 3.1 生产 AI 面试系统已经达到怎样的规模

[Evaluating STT × LLM × TTS Combinations for AI Interview Systems](https://arxiv.org/abs/2507.16835) 使用来自生产系统 Zara 的超过 5000 场 AI 主持面试。系统平均每天执行约 1500 场、每场 20 到 30 分钟，支持 15 种以上语言。这说明自动语音面试不是概念演示，而是已经发生的 L2 级流程替代。

论文比较五种 STT、LLM 和 TTS 组合。Google STT + GPT-4.1 + Cartesia TTS 在其数据中表现最好；GPT-4.1 相比 GPT-4o 的 conversational quality 为 `8.66 vs 8.33`，technical quality 为 `8.48 vs 8.32`，候选人评分为 `4.53 vs 4.29`。更值得注意的是，系统差异主要受 STT 影响，其次才是 LLM，TTS 的贡献较小但稳定。这意味着生产面试中，口音、术语和转录错误可能比“换一个更强推理模型”更直接地决定体验和评估。

该研究最关键的结果不是哪个组件赢，而是 **LLM-as-a-Judge 指标与用户满意度的相关性多数低于 `0.11`**。自动 judge 内部的多个质量分数彼此高度相关，却难以解释候选人真实体验；soft-skill 分数与用户评分的相关性相对更高，约为 `0.53`。这说明“自动评价很一致”并不等于“评价了用户在意的东西”。

### 3.2 PolyInterview：数字人面试可以做得完整，但追问仍是瓶颈

[PolyInterview](https://arxiv.org/abs/2607.10310) 代表更完整的模拟面试系统：它根据职位描述和简历生成多阶段问题，使用口型同步数字人进行语音互动，根据回答动态追问，并给出文本、音频和视频层面的评估。

论文报告系统有 101 个账户、1564 次 session、7665 个生成问题，其中 217 次完成完整 session；日志包含 1274 个已回答主问题、342 个追问 turn 和 1425 条评分记录。90.9% 的问题集合覆盖五阶段流程，82.4% 的职位匹配实验将对应 JD 排在第一位。

十名 HR、语言评价和沟通领域专家对输出进行评分：问题计划为 `4.62/5`，追问为 `3.68/5`，反馈报告为 `3.74/5`。更细的分析显示，追问相关性较强，但 answer dependence 为 `3.28`、diagnostic depth 为 `3.13`；反馈建议具有可操作性，但部分改写引入了回答中不存在的内容，faithfulness 只有 `2.75`。

这组结果很符合当前能力边界：AI 很擅长根据 JD 和 CV 生成结构完整的问题，也可以稳定提供练习环境；但真正像优秀面试官一样抓住候选人回答中的矛盾、空白与关键细节，仍明显更难。

### 3.3 ReqElicitGym：会问很多问题，不代表问到了隐含需求

招聘面试不是唯一的 interview。产品经理、软件工程师和咨询人员也需要通过对话发现用户没有主动说出的需求。[ReqElicitGym](https://arxiv.org/abs/2602.18306) 将这种“访谈能力”做成 101 个网站开发场景，用 oracle user 隐藏一组 implicit requirements，并测量模型能否通过多轮澄清和 probing 把它们问出来。

它提出三类指标：Implicit Requirements Elicitation Ratio 衡量找回了多少隐含需求；Effective Strategy Ratio 衡量 clarification / probing 是否真正获取新信息；Turn-discounted Key Question Rate 奖励尽早问到关键问题。

七个模型的结果并不乐观。non-CoT 下最好的是 DeepSeek V3.2，IRE 也只有 `0.32`，即平均只获取约三分之一隐含需求。GPT-5.2 平均接近耗尽 20 轮预算，却只有 `0.13` IRE；Claude Opus 4.5 约为 `0.08`。CoT 通常提高关键问题的提问顺序效率并减少轮次，却没有稳定提高信息覆盖率。模型普遍偏爱宽泛 probing，较少提出精确 clarification。

这直接挑战“LLM 很会聊天，所以能替代访谈者”的推断。流畅问题和完整信息采集是两种不同能力；停止时机也很重要，太早结束会漏需求，机械地问满 20 轮又会浪费用户时间。

### 3.4 YIELD：信息采集 Agent 正成为独立研究对象

[YIELD](https://arxiv.org/abs/2604.10968) 明确提出 Information Elicitation Agent：不同于被动回答用户问题的助手，它代表招聘方、研究机构、司法或新闻调查等目标，主动从受访者处获取信息。

YIELD 包含 2281 段经过许可的人类对人类访谈、390205 个 turn、约 2625 万 token，覆盖学术访谈、新闻调查、司法对话和口述历史。平均每段对话约 171 个 turn，远长于 MultiWOZ 与 SGD。论文将信息采集建模为有限时域 POMDP，并提出 Conformity、Progression、Turn-Length Ratio 等指标。

这条路线的重要性在于，它不再把采访者当作“回答生成器”，而是当作具有机构目标的 policy。但它也带来一个治理问题：当 Agent 的目标是最大化信息获取时，可能与受访者的隐私、舒适和拒绝权冲突。高信息增益不能成为无限追问的正当理由。

### 3.5 人类监督并不是落后的过渡方案

[Ethics and Social Responsibility in AI-Assisted Interviewing](https://arxiv.org/abs/2606.30980) 使用 Wizard-of-Oz 设计，让 GPT-4o 实时生成 1 到 3 个追问候选，由人类 co-interviewer 选择、修改并说出来。17 名有质性研究经验的参与者提出五类风险：有害或歧视性语言、因分心和漏看非语言信号而损害尊重、数字参与不平等、错误责任不清，以及录音、转录和第三方模型带来的隐私与合规问题。

这项工作没有追求全自动，反而提供了一个现实设计：AI 可以减轻采访者实时组织追问的认知负担，但人类保留关系维护、伦理判断和最终发言控制。在高风险面试中，这种 L1 模式可能长期优于追求完全无人化。

## 4. 客服与企业流程：从会说话到真正完成工单

客服是最可能先出现 L3 Agent 的场景，因为目标、政策、工具和数据库状态通常可以被结构化。但现有结果仍显示，完成一次任务和可靠重复完成任务之间差距巨大。

### 4.1 tau-bench：一次成功不代表可部署可靠性

[`tau-bench`](https://arxiv.org/abs/2406.12045) 模拟零售和航空客服，Agent 与用户多轮对话，同时必须遵守领域政策并调用 API。评价不只看回复，而是比较对话结束后的数据库状态与目标状态。

论文发现包括 GPT-4o 在内的先进 function-calling agent 成功率仍低于 50%。更重要的是，它提出 `pass^k`：同一任务独立运行 k 次，全部成功才算可靠。零售领域的 `pass^8` 低于 25%，说明模型可能偶尔做对，却无法像生产坐席一样稳定执行规则。

[`tau2-bench`](https://arxiv.org/abs/2506.07982) 进一步让用户也能操作共享环境，例如技术支持中用户需要根据 Agent 指示修改路由器设置。实验显示从单控制转为双控制后性能显著下降，因为 Agent 不只要自己推理，还要判断用户是否理解、是否已经执行、执行结果是什么，并在沟通失败时修复流程。这比普通 tool calling 更接近真实人机协作。

### 4.2 中断后的恢复，比“停得快”更接近工作能力

[IHBench](https://arxiv.org/abs/2606.19595) 研究结构化工作流中的 post-interruption recovery。它覆盖客服、医疗预约和账户管理等 10 个企业领域，设置 normal、impatient、correction、topic switch、filler 和 pushback 六类中断，并比较 27 个 audio-language model 配置。

普通 full-duplex benchmark 关注模型是否及时停止；IHBench 继续追问：模型停下以后，能否吸收纠正、回答插话、跳过用户不想听的解释，并回到正确流程节点？结果显示，即使先进模型也有明显缺口。GPT 系列音频模型在 filler interruption 后准确续接原句的通过率只有 `7%–31%`，Gemini 2.5 系列为 `62%–68%`。闭源模型总体更稳，随对话变长的退化速度约比开源模型慢 `3.3×`。

对真实岗位而言，这比 voice naturalness 更重要。用户说“嗯嗯”时从头重念一遍只是体验差；用户纠正邮箱后 Agent 没更新数据库，或者话题切换后丢失原工单状态，就是业务失败。

### 4.3 实时 Agent 为什么不能等工具全部完成再说

[Speculative Interaction Agents](https://arxiv.org/abs/2605.13360) 试图解决工具 Agent 的延迟问题。标准流程等待用户说完、模型推理、调用工具、等待结果，再继续说话，容易产生数秒静默。该工作提出 asynchronous I/O，让模型在等待用户或工具时继续推理；同时用 speculative tool calling 在信息尚未完整时预先发起安全操作。

云端强模型得到 `1.3–1.7×` 加速，Qwen2.5-3B-Instruct 与 Llama-3.2-3B-Instruct 得到 `1.6–2.2×` 加速，代价是轻微准确率损失。其安全设计尤其值得注意：读操作可推测执行，具有不可逆副作用的写操作必须等 final confirmation；未提交的 tool call 可以 modify、remove 或级联取消。

这正是实时工作 Agent 需要的“事务语义”。为了显得快而提前查询订单是合理的，为了显得快而在用户没说完时发送邮件、退款或拒绝候选人则不可接受。

## 5. 医疗对话：受控实验可以超过医生，但不能跳过部署证据

[AMIE: Towards Conversational Diagnostic AI](https://arxiv.org/abs/2401.05654) 是“AI 接替高技能互动工作”最强的公开证据之一。研究使用 149 个临床病例、20 名基层医生和经过验证的 patient actor，进行随机、双盲、交叉式文本问诊比较。

根据专科医生评价，AMIE 在 32 个临床相关维度中的 28 个更优；根据 patient actor 评价，在 26 个维度中的 24 个更优，并展示更强的诊断准确率、病史采集、管理推理、沟通和同理心。

但作者明确说明限制：医生被要求使用自己不熟悉的同步文本聊天，这不是其日常临床工作；场景是标准化 OSCE 风格，而非开放世界患者；研究没有证明模型能在真实医院承担责任、处理检查与随访、识别急症、适应当地流程或面对分布外病例。

因此 AMIE 证明的是：在受控文本诊断对话 episode 中，经过专门训练的 LLM 可以达到甚至超过医生对照表现。它不是“AI 已经可以替代医生”的证据，而是说明高质量交互任务不能再被假定为人类天然不可自动化。

## 6. 计算机工作 Agent：操作能力仍落后于语言表现

如果一个实时 Agent 要接替客服、招聘协调或运营人员，它最终必须在多个软件里执行工作，而不是只生成一份建议。

[OSWorld](https://arxiv.org/abs/2404.07972) 在真实 Ubuntu、Windows 和 macOS 应用中设计 369 个开放任务。初始研究中人类成功率超过 `72.36%`，最佳模型只有 `12.24%`。后续模型持续进步，但 GUI grounding、长流程状态和异常恢复仍是主要瓶颈。[OSWorld-Human](https://arxiv.org/abs/2506.16042) 又强调效率：即使 Agent 最终完成任务，极高的端到端耗时也可能让它无法用于生产。

[CRMArena](https://arxiv.org/abs/2411.02305) 与 [CRMArena-Pro](https://arxiv.org/abs/2505.18878) 将评测带入销售和客户关系管理，覆盖查找客户、更新记录、分析机会与多方协作。它们比通用网页 benchmark 更贴近岗位流程，但仍难以复现真实企业的权限体系、脏数据、组织政策和跨团队责任。

这一证据链再次支持任务级判断：生成面试总结和候选人邮件可能很成熟；在 ATS、日历、邮箱和审批系统之间可靠完成全部招聘运营，仍需要严格边界和人工升级。

## 7. “替代人工”的评测不能只看一个成功率

![Evaluation stack](/images/blog/work-agent-evaluation-stack.svg "图 2：工作型交互 Agent 的评测需要同时覆盖实时性、信息获取、流程结果、可靠性、真人结果、公平与治理。")

一个真正面向岗位代理的 benchmark 至少应包含七层。

**实时交互。** 报告端到端 response latency、stop latency、工具等待期间的静默、打断恢复和 P50/P90/P99，而不是只测模型 TTFT。

**信息采集。** 测关键事实覆盖率、隐含需求找回率、重复问题、无效 probing、确认与纠错。面试官必须知道何时追问，也必须知道何时停止。

**任务结果。** 使用数据库状态、工单状态、日历事件、邮件或审批记录等可执行检查，不只让 judge 看转录文本。

**多次运行可靠性。** 使用 `pass^k`、最差分位数和长尾失败。企业流程需要同一输入多次执行都正确，而不是平均偶尔成功。

**真人结果。** 比较用户满意度、信任、任务耗时、候选人完成率、转人工率和长期业务结果。生产 AI 面试研究中低于 `0.11` 的 judge-user 相关性已经说明自动分数不可替代真人测量。

**公平与可访问性。** 按口音、语言、性别、年龄、残障、网络条件和设备质量分层报告 STT 错误、追问差异、评分差异与放弃率。语音交互中的感知偏差会沿流程放大为机会分配偏差。

**治理与可恢复性。** 测隐私披露、授权、人工接管、申诉、日志可审计性、写操作确认、错误回滚和责任归属。无法安全失败的 Agent，不应被视为可替代人工。

## 8. 招聘面试尤其需要区分“互动质量”和“预测效度”

AI 面试系统通常会给出 communication、confidence、soft skills、technical quality 等分数。但这些标签是否稳定、是否与岗位绩效相关、是否对不同群体公平，是独立于对话自然度的问题。

一个科学的招聘 Agent 评测应采用多阶段设计。

- **内容有效性**：问题是否来自明确的岗位能力模型，而不是模型自由生成的“看起来专业”的问题。
- **结构一致性**：不同候选人获得可比较的核心问题，同时允许有限、可追溯的回答依赖追问。
- **证据忠实性**：每项评价必须链接到候选人的实际回答，不能像 PolyInterview 的部分 polished response 那样引入未表达内容。
- **评分可靠性**：同一回答跨时间、模型版本和随机种子的一致性；多 judge 之间的一致性；人与 AI 的一致性。
- **效标关联效度**：面试分数是否能预测后续工作样本、试用期表现或结构化人工评价，而不是只与另一个 LLM judge 相关。
- **群体公平**：在同等岗位证据下，不同口音、表达风格、语言和辅助技术使用者是否获得相近评价。
- **候选人权利**：明确告知 AI 使用方式，允许请求人工面试、纠正转录、查看评分依据和提出申诉。

如果系统只证明“候选人愿意完成 AI 面试”，它证明的是可用性；如果证明“AI 问题质量高”，它证明的是面试过程；只有进一步证明评分的效度、公平和长期结果，才接近替代人工评价者。

## 9. 从实时模型到 Workforce Agent 的系统架构

![Work agent architecture](/images/blog/work-agent-architecture.svg "图 3：可部署的岗位 Agent 不是单一语音模型，而是实时交互、流程状态、工具事务、政策检查和人工升级组成的系统。")

一个可部署的实时工作 Agent 通常需要以下职责分离。

- **Streaming interaction**：处理语音、视频、屏幕和文本，负责 endpoint、barge-in、speaker / addressee 与低延迟 backchannel。
- **Conversation state**：记录已问问题、已确认事实、未解决冲突、用户目标、情绪风险和当前流程节点。
- **Domain policy**：将企业规则、招聘 rubric、医疗 protocol 或客服 policy 作为显式约束，而不是只放进长 prompt。
- **Planner and tools**：生成可执行 action graph，区分只读查询、可撤销写入和不可逆操作；支持 speculative read、commit、cancel 与 rollback。
- **Evidence and evaluator**：将每个结论链接到对话片段、工具结果和规则，避免凭模型语感产生评分。
- **Human escalation**：根据低置信度、高风险、用户请求、政策例外和重复失败触发接管，并携带结构化上下文而不是让用户从头解释。
- **Audit and monitoring**：记录模型版本、提示、工具调用、延迟、取消、最终状态和申诉结果，支持生产回放与分群审计。

这类架构看起来比一个端到端 omni model 复杂，但复杂性来自真实岗位本身。把所有职责塞进单一模型并不会消除复杂性，只会让错误更难发现和追责。

## 10. 为什么更自然的数字人不一定更适合替代人工

面试和服务产品越来越强调 lip-sync avatar、情绪语音、眼神和人格一致性。这些设计可以提高沉浸感，却也可能增加不恰当信任。用户可能把自然停顿和同理语气误认为系统真正理解了上下文，或者不清楚自己面对的是自动评分系统。

对高风险场景，最重要的不是让 AI 尽可能像人，而是让用户准确理解系统能力、数据用途和控制边界。系统应明确说明：是否录音、是否生成评分、评分是否影响决策、谁能查看数据、何时转人工，以及用户怎样纠正错误。

这也意味着 HCI 评测不能只问“自然吗”和“像真人吗”，还要问“用户能否正确预测系统会做什么”“是否知道如何停止”“是否感到有申诉渠道”“自然外观是否诱发了超过系统真实能力的信任”。

## 11. 未来最值得研究的方向

第一，**从 user simulator 走向真人长期评测**。模拟用户适合可重复 benchmark，但会继承生成模型的语言分布和合作倾向。需要未见用户、不同口音、拒绝回答、含糊表达、对抗行为和真实情绪条件下的研究。

第二，**评测 Agent 何时应该转人工**。当前 benchmark 通常只奖励独立成功，却很少奖励正确升级。生产系统的最佳 policy 可能不是 100% 自动化，而是在少数高风险节点及时交给人类。

第三，**信息采集与用户利益的双目标优化**。YIELD、ReqElicitGym 等推动模型问到更多信息，但未来 reward 还应包括知情同意、最小必要信息、拒绝权、问询负担和关系维护。

第四，**实时工具事务与中断恢复**。语音 stop、流程 state、tool cancellation 和数据库 commit 必须属于同一个事务。用户说“等等”时，停止朗读而继续提交操作不是正确中断。

第五，**模型更新后的持续认证**。STT、LLM、TTS、prompt 或 policy 任一组件变化都可能改变公平性和用户体验。生产面试论文已经显示组件选择影响明显，因此不能只在系统首次上线时做一次评测。

第六，**任务级劳动影响，而不是职业级宣传**。[WORKBank](https://arxiv.org/abs/2506.06576) 通过 1500 名领域工作者、52 名 AI 专家、104 个职业和 2131 个可在计算机上完成的任务，分别评估自动化意愿与技术可行性，并提出从 H1“AI 完全处理”到 H5“持续依赖人类”的 Human Agency Scale。它强调同一职业内部任务差异巨大：有些适合自动化，有些需要平等协作，有些必须由人主导。

## 12. 结论：现在正在替代的是 episode，不是完整职业

研究和生产证据都显示，实时交互 Agent 已经开始接管具体工作 episode：主持 20 分钟面试、完成订单修改、收集需求、安排预约、进行标准化问诊或在 CRM 中更新记录。语音和数字人让这种接管从后台自动化变成用户直接面对的前台服务。

但最可靠的证据仍集中在结构明确、可验证、风险可控的任务。当前模型的主要缺口包括：隐含信息覆盖不足、追问深度有限、多次运行不一致、中断后丢失流程、自动 judge 与真人结果脱节、GUI 和跨应用执行脆弱，以及公平、隐私和责任机制不完整。

所以，我对这一方向的判断是：**实时人机交互正在从“AI 作为界面”走向“AI 作为流程执行者”，接下来真正决定能否替代人工的，不是声音是否更像人，而是系统能否在长尾情况下稳定完成任务、正确请求帮助、尊重用户权利，并对行动结果承担可审计的责任。**
